📝 Skill Authoring — Key Rules

description in YAML frontmatter is the SEMANTIC TRIGGER — write it like a command, not a summary. Model reads it and decides whether to load the skill.

Progressive disclosure: model sees only description at startup. Full content loads on-demand.

Keep skill body under 5,000 tokens. Use HERMES_SKILL_DIR and HERMES_SESSION_ID variables.

🧠 Memory Hierarchy (Know What Loads When)

  • SOUL.md — Identity (~3K tokens, always loaded)
  • MEMORY.md — Facts (~3K tokens, always loaded)
  • USER.md — Preferences (~1K tokens, always loaded)
  • Skills — Zero cost until invoked
  • state.db FTS5 — History, zero until session_search

📡 Multi-Agent: Current Limits

delegate_task has constraints: isolated children, no state sharing, flat dispatch, MAX_DEPTH=2, no crash recovery. A Workflow DAG (dependency-aware task graphs) is planned (Issue #344).

You can already use delegate_task(goal, context, backend='docker') to sandbox untrusted code safely.

🔒 Security Quick Wins

  • Never set GATEWAY_ALLOW_ALL_USERS=true — use explicit allowed_users lists
  • Tirith scans for injection attacks automatically
  • Smart Approvals LLM risk-rates commands before execution
  • Command Approvals can require session-level or always-approve for specific commands

💰 Cost Optimization

  • /usage — monitor spend in real time
  • /compress — summarize history to free context
  • Keep context stable — more cache hits
  • Avoid changing models mid-session
  • Config: auxiliary.compression provider/model/threshold
